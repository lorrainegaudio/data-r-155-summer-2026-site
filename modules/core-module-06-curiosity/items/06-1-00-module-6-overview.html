🐱 Curiosity

This module focuses on curiosity as an analytical habit: asking what the data already contains, what can be clarified, and what patterns appear when you compare groups instead of looking only at individual rows. In Core, you will work with recoding and grouped summaries. You will use ifelse(), case_when(), and mutate() to create clearer variables, then use count(), group_by(), summarize(), and n() to describe grouped patterns.

The main technical move in this module is moving from raw values to more useful analytical results. Recoding helps you turn existing values into labels, categories, or transformed columns. Grouped summaries help you compare categories, count combinations, and compute statistics within groups. Together, these skills let you ask better questions of a dataset instead of accepting the original columns as the only possible structure.

The module follows the usual Core pattern: complete the guided activity, use the supplemental materials only when you need another explanation, complete the Practice Space, and submit the rendered HTML report. The work in this module depends on careful Quarto habits: save the source file, render early, check output, write memos, and make sure your code and interpretation tell the same story.

Learning Objectives

By the end of this module, the successful student will be able to:

  • Use ifelse() to create a new vector with one result for every position.
  • Apply case_when() to recode values using multiple conditional rules.
  • Explain why the order of rules matters when using case_when().
  • Use mutate() to create a new column or overwrite an existing column in a data frame.
  • Interpret checks such as table(), head(), summary(), and colnames() to confirm that recoding worked.
  • Use count() to report frequencies for categorical variables and category combinations.
  • Use group_by() and summarize() together to compute separate summary results for groups.
  • Compute multiple summary statistics, such as means, standard deviations, and row counts, in one grouped summary.
  • Distinguish between a grouping variable and a summary variable in a grouped analysis.
  • Write Quarto memo notes that explain what a recoding or grouped-summary result shows.

Why This Topic Is Important

Real datasets rarely arrive in the exact form needed for analysis. Values may need to be relabeled, grouped, converted, or combined into new variables before they can answer a useful question. Recoding helps you make those changes deliberately instead of editing data by hand or creating unclear one-off fixes.

Grouped summaries are just as important because many analytical questions are comparative. A single average for a whole dataset may hide differences across groups. Once you can group data by category and summarize within those groups, you can ask more useful questions about patterns, differences, and relationships.

This module also strengthens reproducible reporting. Because you are working in Quarto, your code, output, checks, and explanations appear together in one rendered HTML report. That matters because recoding and grouped summaries are easy places to make quiet mistakes. A clear report helps you and your reader see what changed, what was counted, and what the result means.

Strategies for Success

  • For recoding tasks, identify the original column, the new output name, and the rule before writing code.
  • When using case_when(), check the rule order before running the code. Earlier rules are tested first.
  • Use table() after recoding labels so you can see whether the expected categories actually appeared.
  • When using mutate(), be clear about whether you are creating a new column or overwriting an existing one.
  • For grouped summaries, ask two questions before coding: “What creates the groups?” and “What value am I summarizing?”
  • Use .groups = "drop" or ungroup() when the grouped result should no longer behave as grouped data.
  • Render your Quarto file before the end. Do not wait until all tasks are finished to discover a rendering error.
  • Write memo comments that interpret the output, not just comments that repeat the code.

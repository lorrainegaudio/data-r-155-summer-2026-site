🦁 Courage

This module focuses on courage in R work: noticing problems, reading error messages carefully, asking for help clearly, and making deliberate choices when the data are messy. You will continue building R Notebook habits by using Markdown headings, YAML fields, rendering tools, and memo notes to create work that can be checked and shared.

The Core work has two main topics. In Help!, you will practice rendering an R Notebook to HTML, reading the Render tab, finding the first error, and writing better troubleshooting prompts. In Missing Values, you will work with NA, NaN, help files, function arguments, na.rm = TRUE, anyNA(), is.na(), which(), nesting functions, removing missing values, and simple imputation.

This module asks you to slow down when something goes wrong instead of guessing or starting over. The pattern is still familiar: complete the Guided Activity, use supplemental materials when needed, complete the Practice Space, and submit the required file. The important shift is that your work now needs to show how you handled uncertainty, errors, and missing information.

Learning Objectives

By the end of this module, the successful student will be able to:

  • Format an R Notebook with a clear YAML title, date field, Markdown headings, code chunks, and memo notes.
  • Render or knit an R Notebook to an HTML report.
  • Distinguish between Preview and Knit in the RStudio notebook workflow.
  • Use the Render tab to identify the first error when a notebook fails to knit.
  • Write a clear troubleshooting prompt that includes the goal, code, error message, and what has already been tried.
  • Use help files to distinguish required arguments from optional arguments with defaults.
  • Distinguish between NA and NaN in R output.
  • Apply na.rm = TRUE when a summary function needs to ignore missing values.
  • Use anyNA(), is.na(), which(), and sum() to detect, locate, and count missing values.
  • Compare removing missing values with replacing missing values through simple imputation.
  • Use nested functions to solve multi-step problems while keeping the code readable.
  • Explain the difference between a function object such as mean and a function call such as mean().

Why This Topic Is Important

R work does not become reliable just because the code runs once. A notebook may look fine in Preview but fail when rendered, and a calculation may return NA because missing values were not handled intentionally. This module helps you build the habit of checking the actual workflow, not just the visible result.

Errors and missing values are normal parts of data work. The problem is not that they happen; the problem is moving past them without understanding what they mean. Reading the first error, checking function arguments, locating missing values, and documenting your choices help prevent quiet mistakes that can change an analysis without warning.

This module builds directly on earlier work with notebooks, vectors, indexing, and transparency. It prepares you for later modules where you will work with larger datasets, imports, cleaning decisions, and visual summaries. Those later tasks require the courage to stop, inspect, explain, and revise instead of assuming the output is correct.

Strategies for Success

  • Knit or render early. Do not wait until the end to find out that the notebook does not build.
  • When rendering fails, read the Render tab and fix the first error before chasing later messages.
  • Keep Markdown headings clear so your notebook is readable as a report, not just a scratchpad.
  • When asking for help, include the goal, the code, the error message, and what you already tried.
  • Use help files to check function arguments instead of guessing what an argument does.
  • When a summary returns NA, check whether missing values are present before changing the code.
  • Use anyNA(), is.na(), and which() for different questions: whether anything is missing, which values are missing, and where they are located.
  • Explain missing-value choices in memo notes. Removing values and imputing values are decisions, not just code tricks.

🤔 Openmindedness

Openmindedness in this module means being willing to compare different ways of solving the same data problem. You will first use base R bracket subsetting to build logical filters with &, |, %in%, !, and relational operators such as ==, !=, <=, and >=. Then you will use dplyr tools such as select(), filter(), and the pipe %>% to make row and column subsetting more readable.

The main Core work asks you to subset data for a clear reason, not just because the code runs. You will use gapminder, mtcars, murders, stars, and temp_carbon to practice choosing rows, choosing columns, combining multiple conditions, checking results, and explaining the logic behind your choices. You will also continue moving from R Markdown notebooks toward Quarto documents, so your workflow includes code, output, and written reasoning in one reproducible file.

The module follows the usual Core pattern: begin with a guided activity, use optional supplemental materials as needed, complete the Practice Space, and submit the required file. The first half focuses on logical operators and base R subsetting. The second half introduces dplyr subsetting and Quarto. By the end of the module, you should be able to choose between base R brackets and dplyr verbs based on what makes the logic clearest for the task.

Learning Objectives

By the end of this module, the successful student will be able to:

  • Use relational operators such as ==, !=, <=, and >= to build logical tests for filtering rows.
  • Apply &, |, %in%, and ! to combine or reverse logical conditions in base R subsetting.
  • Use bracket subsetting with df[rows, cols] to keep, drop, or inspect selected rows and columns.
  • Explain why parentheses matter when combining & and | in the same filtering rule.
  • Create focused data subsets from larger datasets using select(), filter(), and %>%.
  • Distinguish between selecting columns with select() and filtering rows with filter().
  • Interpret verification checks such as unique(), nrow(), range(), sum(), head(), and all() to confirm that a subset matches the intended rule.
  • Document subsetting logic and checks in a reproducible Quarto workflow.

Why This Topic is Important

Subsetting is one of the most common tasks in data analysis. Analysts rarely use every row and every column at once. They usually need to focus on a specific year, region, group, variable, threshold, or condition. This module gives students the tools to translate plain-English rules into precise R code, then check whether the result actually matches the intended rule.

This module also helps prevent a common beginner problem: writing code that runs but does not filter the data correctly. Small logic errors with &, |, %in%, parentheses, or row-versus-column selection can quietly produce the wrong dataset. Learning to inspect ranges, count rows, check unique values, and verify conditions builds the habit of questioning output before trusting it.

The dplyr work matters because it gives students a more readable way to express many subsetting tasks. Base R brackets remain useful for direct row-and-column indexing, but select(), filter(), and pipes often make the intent of the code easier to read. Comparing both approaches builds flexibility, which matters later when students summarize, graph, model, and revise project work.

Strategies for Success

  • Write the rule in plain English before writing the filter. For example: “keep rows from 2010 and only these three regions.”
  • Build complex filters in pieces first. Create separate logical objects such as is_year and is_region before combining them with &.
  • Use parentheses when mixing & and |. Do not assume R will group the logic the way you meant.
  • Use %in% when checking whether a value belongs to a set of allowed choices. It is cleaner than writing several == tests joined with |.
  • Check your subset immediately after creating it. Use tools such as nrow(), names(), unique(), range(), and all().
  • Remember the difference between rows and columns: filter() changes rows, while select() changes columns.
  • In Quarto, keep memo text outside code chunks so your explanation reads like part of the report, not like hidden code comments.
  • Render or run your document before submitting so you catch missing packages, object-order problems, or code chunks that depend on old Environment objects.
